## MCP Context Bloat Research (2026-03-08/09)

### Confluence Pages
- Main: https://miguelms.atlassian.net/wiki/spaces/AIM/pages/164322 (page_id=164322)
- Deep Dive subpage: https://miguelms.atlassian.net/wiki/spaces/AIM/pages/164466 (page_id=164466)

### Key Findings

**Hard numbers (confirmed):**
- 8 MCP servers, 224 tools = ~66,000 tokens of tool schemas per request
- Claude Sonnet 200K context → 33% wasted before a single message
- smartmcp reduces this to ~1,600 tokens (97% reduction)

**Production solution found — smartmcp:**
- Repo: github.com/spak2005/smart-mcp (6 stars, ~12 days old)
- FAISS + sentence-transformer proxy MCP server
- Exposes single `search_tools` tool; surfaces top-k schemas on demand
- Two-phase: discover → call
- Limitation: needs Python + FAISS infra; O(n) at startup for indexing

**Primitive Groups proposal (MCP spec Discussion #1567):**
- Author: @scottslewis, Sep 29 2025, 100+ comments, not adopted
- Proposes nested ToolGroup structure for Tools/Resources/Prompts
- Stuck on JSON serialization debate (name refs vs copy-on-ref vs JSON Pointers/JRef)
- Critical gap: does NOT reduce context by itself — only adds labels/taxonomy
- Clients still need to implement filtering logic

**OpenClaw architecture — the twist:**
- OpenClaw does NOT natively register MCP tools into LLM context
- MCP accessed via mcporter CLI proxy → model sees one tool with subcommands
- Tool schemas only appear when agent explicitly runs `mcporter list <server>`
- This is accidentally the CLI hierarchical pattern Miguel described
- Nobody has documented this as a design pattern

**Comparison table:**
- Claude Desktop/Cursor/nanobot: native MCP, all tools at startup, O(n) context cost
- OpenClaw: mcporter CLI proxy, O(1) context cost, schemas on demand
- smartmcp: semantic proxy, O(1) at inference, O(n) at startup
